# AIGC 大模型应用工程师 · 知识库

## 一、RAG（检索增强生成）

### 1.1 什么是 RAG
RAG（Retrieval-Augmented Generation）是将信息检索与自然语言生成相结合的技术框架。在生成回答之前，先从外部知识库中检索相关信息，然后将检索结果作为上下文提供给 LLM，生成更准确、更可靠、更具时效性的回答。

核心公式：回答 = LLM(用户查询 + 检索到的相关知识)

### 1.2 为什么需要 RAG
- 知识截止日期：LLM 训练数据有截止时间，无法获取最新信息
- 幻觉问题：LLM 可能生成看似合理但实际错误的内容
- 领域知识缺失：通用 LLM 缺乏专业领域的私有知识
- 可解释性不足：难以追溯 LLM 回答的信息来源
- 成本高昂：频繁微调 LLM 成本极高

### 1.3 RAG 核心架构
经典 RAG 由三大模块构成：
1. 索引模块（离线）：原始文档 → 文档加载 → 文本分割 → 向量化 → 向量数据库
2. 检索模块（在线）：用户查询 → 查询向量化 → 相似度检索 → 上下文组装
3. 生成模块：检索上下文 + 用户查询 → Prompt 模板 → LLM 生成 → 回答

### 1.4 发展历程
- 2020：Facebook 提出 RAG 原始论文
- 2022：ChatGPT 引爆 LLM 应用，RAG 成为主流方案
- 2023：LangChain、LlamaIndex 框架成熟
- 2024：Advanced RAG（多路召回、重排序、Agentic RAG）
- 2025：GraphRAG、Modular RAG、Agentic RAG 成为主流方向

### 1.5 文档加载
支持：.txt、.md、.csv、.pdf、.docx、.pptx、HTML、JSON、SQL/NoSQL 数据库、REST API

### 1.6 文本分割策略
| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 固定长度分割 | 按固定字符数或 token 数分割 | 通用场景 |
| 递归字符分割 | 按段落、句子、单词层级递归分割 | 保持语义完整性 |
| 语义分割 | 基于语义相似度进行分割 | 高质量需求 |
| 结构化分割 | 按文档结构（标题、段落）分割 | 结构化文档 |
| Agentic 分割 | 使用 LLM 智能决定分割边界 | 复杂文档 |

重叠策略：相邻 chunk 保留 10%-20% 的重叠，避免关键信息被截断在边界处。

### 1.8 检索方式
1. 向量检索：余弦相似度、欧氏距离，使用 HNSW/IVF/PQ/DiskANN 等 ANN 算法
2. 关键词检索：BM25（经典算法，IDF 加权词频匹配）
3. 混合检索：向量 + BM25，使用 RRF（Reciprocal Rank Fusion）融合
4. 多路召回：同时进行语义向量、关键词、稀疏向量（SPLADE）、图检索（GraphRAG）
5. 重排序：Cohere Rerank、BAAI/bge-reranker-large、cross-encoder/ms-marco

### 1.9 向量数据库
| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| Milvus | 开源，分布式，高性能 | 大规模生产环境 |
| Pinecone | 托管服务，易用 | 快速原型开发 |
| Chroma | 轻量级，易集成，零运维 | 本地开发/小型项目 |
| Qdrant | Rust 编写，高性能 | 高性能需求 |
| Faiss | Meta 开源，纯向量检索 | 研究/高性能计算 |
| pgvector | PostgreSQL 扩展 | 已有 PG 基础设施 |

### 1.10 查询优化技术
- HyDE（Hypothetical Document Embeddings）：生成假设文档，用假设文档检索
- 查询扩展：同义词扩展，LLM 改写
- 多查询生成：生成多个相关查询并行检索
- 上下文压缩：Map-Reduce、Stuff、Refine、LLM 摘要

### 1.11 高级 RAG 模式
- Self-RAG：LLM 自主决定是否需要检索，插入反思 token 自我评估
- Corrective RAG：检索结果不佳时，自动转向网络搜索
- Agentic RAG：将 RAG 与 Agent 框架结合，支持复杂任务规划和子查询
- Modular RAG：可插拔模块，动态路由

### 1.12 RAG 评估指标
- Context Precision：检索上下文中相关 chunk 的比例
- Context Recall：回答引用的信息占所有相关 chunk 的比例
- Faithfulness：生成内容与检索内容的一致性
- Answer Relevance：回答与问题的相关性
- 评估框架：RAGAS、ARES、TruLens

### 1.13 RAG 代码示例（LangChain）
```python
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA

# 文本分块
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(documents)

# 向量存储
vectorstore = Chroma.from_documents(chunks, embedding_model, persist_directory="./chroma_db")

# 检索 + 生成
qa_chain = RetrievalQA.from_chain_type(llm=llm, retriever=vectorstore.as_retriever(search_kwargs={"k": 5}))
result = qa_chain({"query": "什么是 RAG?"})
```

### 1.14 当前挑战与未来方向
挑战：多跳推理、长文档处理、多模态 RAG、实时性、幻觉问题、成本优化
方向：Agentic RAG、GraphRAG 普及、多模态 RAG、联邦 RAG、自适应 RAG、RAGaaS


## 二、LoRA/QLoRA 微调

### 2.1 LoRA 核心原理
LoRA（Low-Rank Adaptation）由 Hu 等人 2021 年提出，用极少量的可训练参数达到接近全量微调的效果。

核心假设：微调时的权重更新 ΔW 可以用两个低秩矩阵的乘积近似：ΔW = B × A
- B 维度为 d×r，A 维度为 r×k
- r（秩）远小于 d 和 k，通常 r=8~128
- 前向计算：h = W₀x + (B×A)x
- 训练时冻结 W₀，仅训练 B 和 A
- 可训练参数从 d×k 骤降到 r×(d+k)，通常仅占原模型的 0.01%~0.5%

### 2.2 LoRA 超参数选择
| 超参数 | 推荐值 | 说明 |
|--------|--------|------|
| 秩 r | 4~128 | 简单任务 4-8，通用指令微调 16-32，复杂领域 64-128 |
| lora_alpha | 通常 2×r | 缩放因子，实际系数 = alpha/r |
| lora_dropout | 0.05~0.1 | 防止过拟合，小数据集调高到 0.1-0.2 |
| target_modules | q_proj/k_proj/v_proj/o_proj + gate/up/down_proj | 覆盖越多层效果越好 |
| learning_rate | 1e-4 ~ 2e-4（LoRA），2e-4 ~ 5e-4（QLoRA） | QLoRA 可稍高 |
| lr_scheduler | cosine | 余弦退火最稳定 |

### 2.3 QLoRA 三项核心技术
1. 4-bit NormalFloat（NF4）：针对权重正态分布设计的理论最优 4-bit 数据类型
2. 双重量化（Double Quantization）：对量化常数再次量化，每参数节省 0.37bit
3. 分页优化器（Paged Optimizers）：GPU 显存峰值时自动分页到 CPU，避免 OOM

### 2.4 不同方案显存对比
| 模型大小 | 全量微调（FP16） | LoRA（FP16） | QLoRA（NF4） |
|----------|-----------------|--------------|--------------|
| 7B | ~120GB | ~28GB | ~6GB |
| 13B | ~220GB | ~52GB | ~10GB |
| 70B | ~1.2TB | ~280GB | ~48GB |

### 2.5 PEFT 库关键操作
- 保存适配器：`peft_model.save_pretrained("./lora-adapter")`，7B 模型仅 30-100MB
- 加载适配器：`PeftModel.from_pretrained(base_model, "./lora-adapter")`
- 合并权重：`model.merge_and_unload()`，推理无额外开销
- 支持方法：LoRA、QLoRA、Adapter、Prefix Tuning、BitFit、IA3

### 2.6 完整 QLoRA 微调代码模板
```python
from transformers import AutoModelForCausalLM, BitsAndBytesConfig, TrainingArguments
from peft import LoraConfig, prepare_model_for_kbit_training, get_peft_model
from trl import SFTTrainer

# 4-bit 量化配置
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

# 加载量化模型
model = AutoModelForCausalLM.from_pretrained(model_name, quantization_config=bnb_config, device_map="auto")
model = prepare_model_for_kbit_training(model)

# LoRA 配置
lora_config = LoraConfig(r=16, lora_alpha=32, lora_dropout=0.05,
    target_modules=["q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"])

model = get_peft_model(model, lora_config)

# 训练
trainer = SFTTrainer(model=model, args=training_args, train_dataset=dataset,
    tokenizer=tokenizer, max_seq_length=2048)
trainer.train()
```

### 2.7 数据集格式（Alpaca 格式）
```
### Instruction: {任务指令}
### Input: {输入内容，可选}
### Response: {输出内容}
```

数据质量 > 数据数量。1000-10000 条高质量数据远好于 10 万条低质量数据。

### 2.8 训练监控关键指标
- 训练损失：稳步下降正常，快速下降后进入平台期可能过拟合
- 验证损失：与训练损失的 gap 扩大说明过拟合
- 梯度范数：突然飙升说明训练不稳定
- GPU 显存：跟踪占用，避免 OOM

### 2.9 常见问题解决
| 问题 | 解决方案 |
|------|----------|
| 灾难性遗忘 | 降低秩到 8、降低学习率到 1e-5、混入 10-20% 通用数据 |
| 小数据集过拟合 | 数据增强 2-3 倍、调大 dropout 到 0.1-0.2、减少 epoch |
| QLoRA 性能下降 | 计算精度设 bfloat16、调大秩到 32-64、合并权重到 FP16 推理 |

### 2.10 LoRA 实际微调经验
- 在 A800 GPU 上使用 LLaMA-Factory QLoRA 微调 Qwen3-8B
- 训练参数：r=64, alpha=128, lr=5e-5, batch_size=4, 3 epochs
- 适配器大小仅 ~100MB，合并后推理效果接近全量微调
- 推理部署：可通过 Ollama（Modelfile）或 vLLM 加载微调后的模型


## 三、vLLM 高性能推理

### 3.1 核心设计理念
vLLM 是专为 LLM 设计的高吞吐量、低内存推理引擎。三大核心机制：PagedAttention、Continuous Batching、KV Cache 量化。

### 3.2 PagedAttention（分页注意力）
问题：传统 KV Cache 为每个序列分配连续显存块，序列长度不一导致严重显存碎片化。

核心设计：将 KV Cache 切分为固定大小的"页"（Page），通过页表管理映射关系。类似操作系统的虚拟内存分页机制。
- 参数 --page-size：默认 16，短序列调小（8），长序列调大（32）

### 3.3 Continuous Batching（连续批处理）
问题：传统静态批处理要求批次内所有序列同时完成推理，短序列需等待长序列。

核心设计：动态接收/处理请求，序列生成完后立即替换为新请求，GPU 始终保持高利用率。类似 CPU 的时间片轮转调度。
- --max-num-batched-tokens：批次最大 token 数，高吞吐调大（4096），低延迟调小（1024）
- --max-num-seqs：批次最大序列数，短序列调大（128），长序列调小（32）

### 3.4 KV Cache 量化
| 量化方式 | 显存节省 | 精度损失 | 启用方式 |
|----------|----------|----------|----------|
| AWQ（4-bit） | ~75% | 极低 | --quantization awq |
| GPTQ（4-bit） | ~75% | 低 | --quantization gptq |
| FP8（8-bit） | ~50% | 可忽略 | --kv-cache-dtype fp8 |

### 3.5 性能对比（Llama2-7B + RTX 4090）
| 指标 | 基础推理 | 调优后 | 提升 |
|------|----------|--------|------|
| 显存占用 | ~15GB（FP16） | ~5GB（AWQ+FP8 KV） | 降低 67% |
| 吞吐量 | ~50 tokens/s | ~200 tokens/s | 提升 4 倍 |
| 延迟 | ~2.5 ms/token | ~1.2 ms/token | 降低 50% |

### 3.6 部署配置示例
```bash
# 基础推理
vllm serve ./model --port 8000 --host 0.0.0.0

# 高性能推理（AWQ量化 + FP8 KV Cache）
vllm serve ./model-awq --port 8000 \
  --quantization awq \
  --kv-cache-dtype fp8 \
  --max-num-batched-tokens 4096 \
  --max-num-seqs 64 \
  --gpu-memory-utilization 0.95
```

### 3.7 核心架构组件
1. PagedAttention：KV Cache 分页管理，解决显存碎片化
2. Continuous Batching：动态批处理，最大化 GPU 利用率
3. KV Cache 量化：支持 GPTQ/AWQ/FP8，大幅降低显存
4. Tensor Parallelism（TP）：模型参数拆分到多 GPU
5. GPU 显存调度优化：自动管理显存分配，优先使用 HBM

### 3.8 Tensor Parallelism
```bash
# 2 块 GPU 做张量并行
vllm serve model --tensor-parallel-size 2
```
- TP 数 ≤ GPU 数量，需匹配模型层数
- 单 GPU 场景无需 TP，反而增加通信开销

### 3.9 硬件要求
- GPU：NVIDIA A10/A100/H100/RTX 3090+，需 CUDA 11.8+
- 7B 模型（FP16）：至少 10GB 显存，4-bit 量化后可降至 ~5GB
- CPU/内存：至少 8 核 16GB，内存 ≥ GPU 显存
- 不支持的 GPU：GTX 1080 等 Kepler 架构

### 3.10 采样策略
| 场景 | 参数 | 目的 |
|------|------|------|
| 高吞吐 | temperature=0, top-p=1.0 | 关闭随机性 |
| 低延迟 | max-tokens=128 | 限制生成长度 |
| 极速模式 | use-sampling=False | 直接贪心解码 |

### 3.11 常见问题
| 问题 | 原因 | 解决 |
|------|------|------|
| OOM | 批处理参数过大 | 减小 max-num-batched-tokens，开启量化 |
| 吞吐未达预期 | 未开 Continuous Batching | 确认动态批处理开启 |
| CUDA error | GPU 架构不支持 | 更换 A10/A100/RTX 3090+ |


## 四、AI Agent

### 4.1 Agent 核心概念
AI Agent 是能够自主理解任务、选择工具、执行多步骤操作的智能体。

三大核心能力：
1. 大模型接入：接入 LLM 提供智能推理基础
2. 工具决策：根据用户意图自主选择和调用工具
3. 任务管理：驱动多步骤任务的规划、执行和状态追踪

Agent vs MCP：Agent 可以独立完成任务，MCP（Model Context Protocol）更像是聚合接口（USB Hub）。

### 4.2 ReAct 框架
ReAct = Reasoning（推理）+ Action（行动），是 Agent 的核心执行机制。

执行循环：
1. 用户输入 → 动态工具检索 → 生成 Prompt → LLM 推理
2. 解析输出 → 需要行动？→ 选择并执行工具 → 观察结果 → 更新状态 → 回到步骤 1
3. 最终答案 → 返回结果给用户

最大推理次数由 max_iterations 控制，建议设 5-10。

### 4.3 ReAct 多轮推理示例
以查询"最贵订单"为例：
- 第 1 轮：LLM 判断需要先了解表结构 → 调用 table_schema 工具
- 第 2 轮：获得表结构后，生成 SQL → 调用 sql_query 工具
- 第 3 轮：获得查询结果，判断已得最终答案 → 返回 AgentFinish

### 4.4 LangChain Agent 实现
三大能力实现：
1. 能力接入：使用 LangChain 内置工具包接入 LLM、连接数据库
2. 工具决策：API 封装 → Tool 注册 → 工具查找。复杂场景先 Embedding Top-K 预筛选，再 LLM 最终决策
3. 任务管理：Prompt Template → LLM 交互 → Output Parser → Agent 调用 → ReAct 循环

### 4.5 Tool/Function Calling 机制
```python
from langchain_core.tools import tool

@tool
def multiply(first_int: int, second_int: int) -> int:
    """两个整数相乘"""
    return first_int * second_int

@tool
def add(first_add: int, second_add: int) -> int:
    """两个整数相加"""
    return first_add + second_add

tools = [multiply, add]
```

### 4.6 Agent 类型
| 类型 | 特征 | 适用场景 |
|------|------|----------|
| Zero-shot ReAct | 仅依赖工具描述推理 | 通用任务 |
| Conversational Agent | 带对话记忆 | 多轮对话 |
| Structured Agent | 结构化输出 | 需要特定格式 |
| OpenAI Functions | 使用原生 Function Calling | OpenAI 模型 |
| Tool Calling Agent | 使用 Tool Calling API | 多工具场景 |

### 4.7 主流 Agent 框架
| 框架 | 特点 | Stars |
|------|------|-------|
| LangChain | 推出最早，工具包最丰富，学习资源多 | 110K+ |
| AutoGen | 微软开源，Azure 生态优势 | 40K+ |
| CrewAI | 多 Agent 协作，角色分工 | 30K+ |
| Google ADK | 最新推出，支持多 Agent 交互 | 新兴 |

### 4.8 Agent 开发最佳实践
- temperature=0：确保推理结果的确定性（强烈建议）
- 工具描述用自然语言清晰说明功能、参数、返回值
- 复杂工具选择先用 Embedding 预筛选 Top-K
- max_iterations 设 5-10 防止无限循环
- 添加安全防护：SQL 注入检测、权限校验、危险操作拦截


## 五、Prompt Engineering

### 5.1 核心概念
Prompt Engineering（提示词工程）是与 LLM 高效交互的核心技能，通过精心设计输入来引导模型生成高质量输出。

### 5.2 Zero-shot vs Few-shot
| 方式 | 定义 | 适用场景 |
|------|------|----------|
| Zero-shot | 直接提问，无示例 | 简单问答、快速验证 |
| Few-shot | 提供 2-5 个示例 | 特定格式、分类任务 |
| Few-shot CoT | 提供带推理过程的示例 | 复杂推理任务 |

### 5.3 指令设计六要素
1. 角色定义：明确 AI 的角色身份
2. 任务描述：清晰说明具体任务
3. 输出格式：指定结构和格式要求
4. 约束条件：设定边界限制
5. 参考示例：提供期望输出的样例
6. 评估标准：说明什么样的输出算好

### 5.4 Chain of Thought（思维链）
引导模型逐步推理，三种方式：
- 基础 CoT：加"请逐步思考"
- Few-shot CoT：提供带推理过程的示例
- 结构化 CoT：使用 <thinking><answer> 标签分隔

### 5.5 Self-Consistency
多次采样不同推理路径（利用 Temperature>0），选最一致的答案。适用于数学、逻辑推理等高可靠性场景。

### 5.6 Prompt Chaining
复杂任务分解为多步，每步输出作为下一步输入。优缺点：质量可控、便于调试，但成本增加、延迟累积。

### 5.7 System Prompt 设计
| 组成 | 说明 |
|------|------|
| 角色定义 | AI 的身份和专业领域 |
| 职责范围 | AI 应该做什么 |
| 输出规范 | 格式、语言、风格要求 |
| 限制条件 | AI 不应该做什么 |

### 5.8 参数调优
| 参数 | 作用 | 推荐范围 |
|------|------|----------|
| Temperature | 控制随机性，低=确定，高=多样 | 事实性 0-0.3，创意性 0.7-1.0 |
| Top-p | 核采样，只从累积 p 概率中选词 | 确定性 1.0，创意性 0.9 |
| Max Tokens | 输出最大长度 | 按任务需求 |
| Frequency Penalty | 降低重复词概率 | -2.0 到 2.0 |
| Presence Penalty | 鼓励新话题 | -2.0 到 2.0 |

### 5.9 JSON 结构化输出
四大技巧：明确要求 JSON、提供 Schema 示例、预填充强制 JSON 开头（assistant content 设为 `{`）、设低 Temperature（0-0.3）。

### 5.10 常见错误
1. 过度工程化：System Prompt 又长又啰嗦 → 简洁精准
2. 只说不要什么，不说要什么 → 正面描述期望输出
3. 一次塞入过多要求 → 用 Prompt Chaining 分步
4. 没有设置 Temperature=0 → 生产环境建议设 0 确保稳定性

### 5.11 语境管理策略
| 策略 | 说明 |
|------|------|
| 滑动窗口 | 只保留最近 N 轮对话 |
| 摘要压缩 | 对历史对话摘要后保留 |
| 关键信息提取 | 仅保留任务相关关键信息 |
| 上下文重置 | 话题切换时清空无关历史 |


## 六、Transformer 与深度学习基础

### 6.1 Transformer 架构
Transformer（2017，"Attention is All You Need"）是现代 LLM 的基石架构。摒弃 RNN 的循环结构，完全基于注意力机制。

核心组件：
1. Self-Attention：每个词关注输入序列中的所有词，捕获长距离依赖
2. Multi-Head Attention：多个注意力头并行计算，不同头关注不同语义层面
3. Feed-Forward Network（FFN）：两层全连接 + 激活函数，引入非线性
4. Layer Normalization：稳定训练，加速收敛
5. Positional Encoding：注入位置信息（Transformer 本身不感知顺序）
6. Residual Connection：残差连接，缓解梯度消失

### 6.2 注意力机制公式
Attention(Q, K, V) = softmax(QK^T / √d_k) × V
- Q（Query）：查询向量
- K（Key）：键向量
- V（Value）：值向量
- √d_k：缩放因子，防止点积过大导致 softmax 梯度消失

### 6.3 LLM 训练三阶段
1. 预训练（Pre-training）：海量无标注文本，自监督学习（预测下一个 token）。耗时数月，成本百万级
2. 指令微调（Instruction Tuning/SFT）：高质量问答对，让模型学会遵循指令。耗时数小时到数天
3. 人类反馈强化学习（RLHF）：人类偏好数据，对齐人类价值观。PPO/DPO 算法

### 6.4 关键技术概念
- Tokenization：将文本切分为 token（BPE/WordPiece/SentencePiece）
- Embedding：将 token 映射为稠密向量
- 上下文窗口（Context Window）：模型一次能处理的最大 token 数。GPT-4o 128K，Claude 200K
- 推理优化：KV Cache（缓存已计算的 Key/Value，避免重复计算）、FlashAttention（IO 感知的精确注意力算法）、Speculative Decoding（小模型预测 + 大模型验证加速解码）
