# 课程助手 — 技术方案确认文档

> 项目：AI Engineer Mentor（课程助手）
> 日期：2026-06-17（修订版 v0.4.0，前后端对接完成）
> 阶段：阶段一 — 技术方案设计

---

## 技术选型总览

```
前端：Gradio 6.18
  ↓ HTTP/SSE (httpx.stream)
后端：FastAPI + LangChain
  ↓
AI 模块：DeepSeek 云端 API + instructor-xl（本地嵌入） + ChromaDB
  ↓
数据层：PyMuPDF/python-docx → 文件解析 → SQLite（会话管理）
```

---

## 决策点 1：后端框架 — FastAPI vs Flask

| 维度 | FastAPI | Flask |
|------|---------|-------|
| 异步支持 | 原生 async/await | 需额外装 Quart |
| 性能 | 高性能，接近 Node.js | 同步模式，并发弱 |
| 自动文档 | Swagger UI 自带 | 需手动加插件 |
| 数据校验 | Pydantic 自动 | 需手动写 |
| 与 LangChain 配合 | langchain-serve 支持好 | 需自己封装 |

**✅ 选择：FastAPI**

课程助手的 RAG 问答链路有多个异步 IO（调用云端 API、查向量库），FastAPI 原生异步是刚需。LLM 调用通过 `asyncio.to_thread` 放入线程池执行，避免阻塞事件循环。

---

## 决策点 2：向量数据库 — ChromaDB vs Milvus vs Pinecone

| 维度 | ChromaDB | Milvus | Pinecone |
|------|----------|--------|----------|
| 部署复杂度 | `pip install` 一行 | 需 Docker | 云服务 |
| 适合场景 | 本地开发 / 小规模 | 百万级 | 企业级 |
| 免费 | ✅ | ✅ | 有限额 |

**✅ 选择：ChromaDB**

课程助手知识库是 AIGC 课程资料，文档量级在几百到几千，ChromaDB 绰绰有余。轻量本地部署，零运维成本。支持 `PersistentClient` 持久化，重启不丢数据。知识库为可重建架构 — 删除 `chroma_db/` 目录后重启后端即自动创建空库。

---

## 决策点 3：嵌入模型 — instructor-xl（本地部署）

| 维度 | instructor-xl（本地） | text-embedding-3-small（API） |
|------|----------------------|------------------------------|
| 调用方式 | sentence-transformers 本地 | API 调用，需联网 + Key |
| 速度 | 首次加载慢，后续快 | 快（OpenAI 托管） |
| 中文效果 | 好 | 较好 |
| 成本 | 免费 | 按 token 计费 |

**✅ 选择：instructor-xl（本地部署）**

纯本地向量化，零外部依赖，与 ChromaDB 深度集成。通过 `embedding.py` 共享模块统一嵌入逻辑，LangChain 和 ChromaDB 原生接口共用同一个 SentenceTransformer 实例。

---

## 决策点 4：LLM 推理 — DeepSeek 云端 API（修订）

| 维度 | DeepSeek 云端 API | Ollama 本地 (Qwen2.5:3B) |
|------|-------------------|--------------------------|
| 模型能力 | DeepSeek-Chat（671B MoE） | Qwen2.5:3B（3B 参数） |
| 硬件需求 | 无（云端推理） | M1 16GB RAM |
| 回答质量 | 高（接近 GPT-4） | 中等 |
| 成本 | ￥1/百万 token | 免费 |
| 部署 | 仅需 API Key | 需装 Ollama |

**✅ 选择：DeepSeek 云端 API（deepseek-chat）**

- 回答质量远超本地 3B 模型，更适合课程答疑场景
- 无硬件限制，兼容 OpenAI API 格式，通过 `langchain-openai` 调用
- 成本极低（￥1/百万 token），学习阶段几乎免费
- 配合 `asyncio.to_thread` 异步调用，不阻塞 FastAPI

---

## 决策点 5：文件解析

| 文件类型 | 解析工具 | 原因 |
|----------|----------|------|
| PDF | PyMuPDF（fitz） | 速度快、中文友好、保留格式 |
| Word | python-docx | 原生 docx 结构解析 |
| PPT / 其他 | Unstructured | 兜底通用解析库 |
| 图片 OCR | 按需扩展 | 暂不纳入 MVP |

**✅ 选择：PyMuPDF + python-docx**

覆盖课程资料的主流文件格式。文件上传限制 50MB，解析后自动分块写入 ChromaDB。

---

## 决策点 6：文本分块 — RecursiveCharacterTextSplitter

改用 LangChain 的 `RecursiveCharacterTextSplitter` 替代手写字符切片：

| 维度 | RecursiveCharacterTextSplitter | 手写字符切片 |
|------|------|-----------|
| 分隔符层级 | `\n\n → \n → 。→ ！→ ；→  ` | 无 |
| 语义完整性 | 高（优先在自然断点切分） | 低（可能在句子中间截断） |
| 参数可配 | chunk_size / overlap / separators | 固定 |

**✅ 选择：RecursiveCharacterTextSplitter**（chunk_size=500, overlap=50）

---

## 决策点 7：前端框架 — Gradio vs Streamlit

| 维度 | Gradio | Streamlit |
|------|--------|-----------|
| 聊天组件 | gr.Chatbot 原生支持 | 需第三方库 |
| 多会话 | gr.State 管理方便 | 需 session_state |
| 生态 | HuggingFace 家族 | 独立项目 |
| CSS 定制 | elem_id + CSS 注入 | 受限 |

**✅ 选择：Gradio 6.18**

已用 Gradio 完成完整聊天界面（多会话管理 + 日期分组 + 文件上传），支持 session_id 与后端实时同步。前端通过 `httpx.stream()` 对接 FastAPI SSE 流式接口。

---

## 决策点 8：业务数据库 — SQLite

| 维度 | SQLite | MySQL/PostgreSQL |
|------|--------|-------------------|
| 部署 | 单文件零配置 | 需数据库服务 |
| 并发 | 中等 | 高 |
| 适用场景 | 本地/小团队 | 生产环境 |
| Python 支持 | 标准库内置 | 需安装驱动 |

**✅ 选择：SQLite**

存储会话历史（最近 10 轮）和用户名记忆。前端每个会话生成 UUID 作为 session_id，后端通过 SQLite 持久化，重启不丢失。

---

## 决策点 9：意图路由 — 嵌入向量余弦相似度

| 维度 | 嵌入向量相似度 | LLM Function Calling |
|------|---------------|---------------------|
| 延迟 | ~0.1s（本地计算） | ~1-3s（云端 API 调用） |
| 准确性 | 中等 | 高 |
| 成本 | 免费 | 消耗 token |

**✅ 选择：嵌入向量余弦相似度 + 会话级缓存**

用 instructor-xl 预计算课程锚点和闲聊锚点的向量，查询时比较用户消息与两组锚点的最大余弦相似度，选得分高的一侧路由。封装为 `IntentRouter` 类，增加 `route_cached(session_id, msg)` 方法 — 同一会话只计算一次嵌入，后续直接用缓存，避免每轮对话都跑 7-15s 的嵌入计算。

## 前后端对接方案（v0.4.0 新增）

```
Gradio 前端 (step8.py)          FastAPI 后端 (8001)
     │                                │
     ├── POST /chat/stream ──→ SSE 流式响应
     │   {session_id, message}        │
     │                                ├── 意图路由（缓存）
     │                                ├── RAG 检索（ChromaDB Top-3）
     │                                ├── DeepSeek 流式生成
     │←── data: {token, done} ──────┤
     │                                │
     ├── POST /upload ─────────→ 文件解析 → 分块 → 入库
     │   {file}                       │
     │←── {status: "ok", chunks} ──┤
```

前端使用 `httpx.stream()` 逐行读取 SSE 事件（`data: {json}\n\n`），解析 token 累积后 yield 给 Gradio Chatbot 实现流式显示。后端未启动时前端显示友好错误提示。
