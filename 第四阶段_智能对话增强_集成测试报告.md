### 2.2 完整请求链路（第四阶段后）

```
用户消息
  │
  ├─ 1. ContextMemory.build_context(history)
  │      ├─ 最近 6 条 → 原文保留（精确上下文）
  │      └─ 更早消息 → 提取关键事实（用户名/主题/待解决问题）
  │            └─ 生成 "【之前的对话记忆】用户叫小明。讨论过LoRA、vLLM…"
  │
  ├─ 2. IntentRouter.route(msg, recent_history)
  │      └─ 5 类意图分类（LLM + k-NN 回退 + 快速规则）
  │
  ├─ 3. ComplexityAnalyzer.analyze(msg)
  │      ├─ 问候/确认/短消息 → simple（规则，0延迟）
  │      ├─ 多步骤/对比/长消息 → complex（规则，0延迟）
  │      └─ 模糊 → LLM 轻量判断兜底
  │
  ├─ 4. [复杂问题] TaskPlanner.plan(msg, intent)
  │      ├─ 对比类 → 3步：检索资料 → 对比分析 → 给出建议
  │      ├─ 教程类 → 2步：检索教程 → 整理步骤
  │      └─ 搜索类 → 2步：搜索信息 → 整理总结
  │
  ├─ 5. RAG/搜索/闲聊 生成回答
  │      ├─ memory_summary 注入 system prompt
  │      └─ 复杂问题输出执行计划（📋 步骤1 → 步骤2...）
  │
  └─ 6. FollowUpRecommender.recommend()
         ├─ 回答中提取技术关键词 → 匹配话题模板
         ├─ 内容类型识别（对比/教程）→ 专用模板
         └─ 非技术内容 → LLM 兜底生成
              └─ SSE done 事件 / JSON response 附带 2-3 个推荐
```

## 三、各模块实现详情

### 3.1 模块1：对话历史管理与上下文记忆

**解决的问题**：用户连续问问题时，系统不能每次都像第一次见到用户一样。

**实现方案**：滑动窗口 + 轻量事实提取（混合方案）

```
原始历史 20 条
  → ContextMemory.build_context(history, recent_window=6)
    → 最近 6 条：原文保留，不做任何处理
    → 更早 14 条：提取关键事实
        • 用户名："我是小明" → user_name=小明
        • 技术主题：含 LoRA/vLLM/RAG 等关键词 → topics
        • 未解决问题："还有一个问题…" → pending
    → 输出："【之前的对话记忆】用户叫小明。讨论过LoRA、vLLM。"
```

**设计决策**：

| 决策 | 理由 |
|------|------|
| 不用 LLM 摘要 | LLM 调用慢且贵，提取事实用正则匹配即可，0 额外延迟 |
| 窗口大小=6 | 课程助手对话通常不长，6 条覆盖 90% 场景 |
| memory_summary 注入 system prompt | 不修改消息结构，兼容现有 RAG/闲聊/搜索三种生成路径 |

**遇到的问题**：

| 问题 | 解决方案 |
|------|---------|
| `_extract_memory()` 漏写 `return memory` | 返回值 None → `_format_memory` 崩溃。补上 return 语句解决 |
| 原 main.py 中 `history` 变量名冲突 | streaming 端点中 `history` 仍是原始全量，需改为 `recent_history`。逐引用排查修正 |
| `build_casual_prompt` 未同步更新 | 也加上了 `memory_summary` 参数支持 |

**优化方向**：

- 事实提取目前仅限 AIGC 关键词，可扩展为通用 NLP 实体识别
- 可增加"重要消息标记"——用户明确说"记住这个"时持久存储

### 3.2 模块2：意图引导与多轮任务路由

**解决的问题**：用户说一句话后，系统要判断他到底想做什么，规划相应流程，然后进入执行。

**实现方案**：规则初筛 + LLM 深度规划（混合方案）

**复杂度判断流程**：

```
ComplexityAnalyzer.analyze(msg)
  │
  ├─ 规则匹配（0 延迟，覆盖 90%+ 场景）
  │   ├─ 问候/确认/short → SIMPLE
  │   ├─ "先…再…"/"对比…和…"/"分析…建议…" → COMPLEX
  │   ├─ 短消息 + 无技术术语 → SIMPLE
  │   └─ 长消息(>80字) → COMPLEX
  │
  ├─ 规则无法判断 → LLM 轻量判断（max_tokens=10）
  │   └─ prompt: "只输出 simple 或 complex"
  │
  └─ 兜底 → SIMPLE（安全优先）
```

**任务规划示例**：

| 用户输入 | 意图 | 计划步骤 |
|---------|------|---------|
| "LoRA 和 QLoRA 有什么区别？哪个更好？" | course_question | ①检索资料 → ②对比分析各方案 → ③给出选择建议 |
| "怎么用 vLLM 部署大模型？" | course_question | ①检索相关教程 → ②整理步骤和实践建议 |
| "什么是 RAG？" | course_question | ①检索知识库资料 |
| "今天天气怎么样" | web_search | ①搜索最新信息 → ②整理并总结 |

**设计决策**：

| 决策 | 理由 |
|------|------|
| 规则优先 | 覆盖问候/多步骤等高频场景，零延迟，不给 LLM 增加负担 |
| LLM 轻量判断 | max_tokens=10，仅对模糊问题，成本极低 |
| 默认兜底为 SIMPLE | 宁可少规划也不要误判导致用户等待 |
| 计划展示在流式输出中 | 用户能看到执行进度，体验更透明 |

**遇到的问题**：

| 问题 | 解决方案 |
|------|---------|
| 复杂查询 LLM 超时导致 fallback | 网络波动时大查询失败，属于 LLM API 问题，降级兜底已覆盖 |
| 复杂度判断与意图分类串行增加延迟 | 目前 analyze() 是同步规则方法（0延迟），不影响。LLM 判断可后续改为并行 |

**优化方向**：

- 多轮任务状态可持久化到 session，支持"继续下一步""重新执行步骤2"
- 计划执行可与 RAG 检索深度整合——每个步骤自动选择检索策略
- 可根据历史对话动态调整计划（而非每次重新规划）

### 3.3 模块3：后续问题推荐系统

**解决的问题**：回答结束后，推荐 2-3 个相关问题供用户点击追问。

**实现方案**：规则模板 + LLM 兜底（混合方案）

```
FollowUpRecommender.recommend(query, answer, intent)
  │
  ├─ 1. 从 answer 提取技术关键词 → 匹配话题模板
  │      LoRA/微调 → "LoRA 和 QLoRA 的主要区别是什么？"
  │      RAG/知识库 → "如何提升 RAG 的检索准确率？"
  │      vLLM/部署 → "vLLM 部署需要什么配置？"
  │
  ├─ 2. 根据内容类型补充
  │      对比类 → "在实际项目中应该怎么选择？"
  │      教程类 → "有没有更简单的替代方法？"
  │
  ├─ 3. 通用技术模板补充
  │      "能再详细解释一下吗？"
  │      "这个技术有什么局限性？"
  │
  └─ 4. [LLM 兜底] 规则产出 < 3 个 → LLM 生成补充
         prompt: "根据对话生成3个后续问题，每行一个"
```

**前后端集成**：

```
后端:
  SSE done 事件 → {"done": true, "followup": ["Q1", "Q2", "Q3"]}
  JSON response → {"answer": "...", "followup": [...]}

前端 (course_assistant_ui.py):
  SSE 解析 → followup = event.get("followup", [])
  渲染 → "💡 你可能还想问：\n• Q1\n• Q2\n• Q3"
```

**设计决策**：

| 决策 | 理由 |
|------|------|
| 规则模板为主 | 快，零延迟，覆盖面广（7 个话题 × 3 模板 + 通用兜底） |
| LLM 异步兜底 | 非技术内容（天气等）规则无匹配时，LLM 补充 |
| 非流式用 async，流式用 sync | 流式端点延迟敏感，规则推荐足够；非流式用 LLM 兜底质量更高 |
| 每次随机选模板 | 同一话题每次推荐不同问题，避免重复 |

**遇到的问题**：

| 问题 | 解决方案 |
|------|---------|
| 非技术内容推荐为空 | 预期行为。加 LLM 兜底（`recommend_async`）解决，同时保留空返回不崩溃 |
| ChatResponse 模型缺少 followup 字段 | Pydantic 模型新增 `followup: list = []` |
| 前端完全不处理 followup | 在 SSE done 事件中添加解析和渲染逻辑 |
| 旧进程未杀导致代码改动不生效 | 重启前用 `pkill -9` 确认端口释放 |

**优化方向**：

- 前端可渲染为可点击按钮（点击后自动填入输入框并发送）
- 可记录用户点击推荐问题的行为，用于后续推荐排序优化
- 推荐模板可扩展支持更多话题领域

## 六、心得体会

### 6.1 混合方案的价值

本阶段三个模块全部采用「规则 + LLM」的混合架构。实践证明这是正确的选择：

- **规则负责 90% 的常见场景**，零延迟、零成本
- **LLM 负责 10% 的边缘场景**，保证覆盖率和质量
- 两者结合比纯规则更智能，比纯 LLM 更快更省钱

这个模式可以推广到项目的其他模块（如检索策略自适应选择、回复质量自动评估）。

### 6.2 测试先行的好处

前期补了 RAG 测试，后续改 `RAG_TOP_K`、`RRF_K` 时跑一下就知道有没有退化。没有这 17 个测试，调参就是盲调。

### 6.3 前端耦合的代价

模块3 的推荐问题在后端已经生成，但前端要单独改才能显示。前后端分离架构下，新功能需要两端同步更新——这是微服务化后需要注意的成本。

### 6.4 chroma_db 路径依赖

集成测试依赖真实的 ChromaDB 数据和 embedding 模型。如果 `chroma_db/` 被清空或 `instructor-xl/` 被移动，集成测试会失败。后续可考虑为测试提供最小化的 mock 数据。
